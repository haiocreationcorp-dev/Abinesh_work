2. COMIC EDITOR URLs (client/src/utils/comicUrl.js — new file)
================================================================================

- Route changed from /editor/:rawId (bare cuid, e.g. "cmqyuurcf0004...") to a
  slug+id combo, e.g. /editor/the-unicorn-industry-cmqyuurcf0004...
- `comicEditorParam(comic)`: builds the slug from the title, appends the real
  id as the last `-`-separated segment.
- `comicIdFromParam(param)`: recovers the real id by splitting on the last `-`
  (cuid ids contain no dashes, so this is always exact). Backward compatible
  with old bare-id links.
- Wired into DashboardPage.jsx's two navigate() calls (open comic, create
  comic) and ComicEditorPage.jsx's useParams() destructuring.
- Note: the browser address bar itself can't be hidden or blanked by the
  app — this only makes the visible URL more readable, the id is still
  present as a suffix (browsers don't allow JS to erase their own chrome).

10. PRODUCTION DEPLOYMENT — LIGHTSAIL SERVER TROUBLESHOOTING
================================================================================

Context: BharathComic runs on an AWS Lightsail Windows Server 2022 instance,
code at C:\BharathComic, deployed via git pull (no CI/CD), served by Node
via PM2, with nginx optionally fronting port 80 -> proxy to Node on :3000.

Issue #1 — untracked-file pull conflicts:
  `git pull origin main` on the server failed twice with "The following
  untracked working tree files would be overwritten by merge" — first for
  client/public/tool-icons/* + bharath-comics-logo.png (this session's new
  icon files had already been manually copied onto the server outside of
  git), then again for server/uploads/renders/* (cached page-render files,
  hashed filenames + .meta.json sidecars tied to live faceIds).
  FIX (both cases, per explicit user confirmation each time): back up the
  at-risk folders to a timestamped directory outside git first, then force
  git to take the incoming version:
    git fetch origin
    git checkout origin/main -- <conflicting paths>
    git pull origin main

Issue #2 — blank white page after deploy (recurring, twice in this session):
  Browser DevTools Console showed: "Failed to load resource... 500 (Internal
  Server Error)" on the built JS/CSS bundle files. PM2 error log
  (`pm2 logs bharatcomic --err --lines 50 --nostream`) showed repeated
  "[Error] Not allowed by CORS" lines.
  ROOT CAUSE: server/src/index.js's CORS origin-check throws an Error for
  any non-matching Origin header, and that thrown error propagates as an
  uncaught 500 for EVERY request (including static asset requests, not just
  API calls) — matching the pre-existing warning already in
  bharathcomic-lightsail-deploy memory ("this broke twice already"). The
  underlying mismatch was server/.env's CLIENT_URL not exactly matching
  however the site was actually being accessed:
    - First occurrence: accessing http://13.234.72.92:3000 directly while
      CLIENT_URL was set to just http://13.234.72.92 (no port).
    - ALSO found in the same .env: DATABASE_URL's password contained a raw
      un-encoded "@" (BharathComic@2024!), breaking pg_dump's connection
      string parsing ("could not translate host name '2024!@localhost'") —
      needs percent-encoding: BharathComic%402024!
    - Second occurrence: after fixing the above, nginx (which proxies port
      80 -> Node on :3000, stripping the :3000 from what the browser
      shows/sends) started being used again, so CLIENT_URL needed to flip
      back to the no-port form (http://13.234.72.92) to match nginx-proxied
      requests.
  FIX: edit CLIENT_URL (and DATABASE_URL's encoding) in server\.env to match
  the live access URL exactly, then:
    pm2 restart bharatcomic --update-env
  IMPORTANT: a plain `pm2 restart` (no flag) does NOT reload .env — it
  reuses whatever environment PM2 cached from the process's original start.
  This was the actual cause of one "I fixed .env but it's still broken"
  report in this session — always restart with --update-env after any .env
  edit on this box.
  DIAGNOSTIC LOG ADDED (kept in, on purpose): server/src/index.js's CORS
  rejection handler now does
    console.error(`[CORS] Rejected origin: "${origin}" — CLIENT_URL is: "${process.env.CLIENT_URL}"`)
  before throwing, so future mismatches show the exact both-sides values in
  `pm2 logs bharatcomic --err` instead of requiring another debug round.

Issue #3 — nginx stopped proxying (separate incident):
  Symptom: previously http://13.234.72.92 (no port) worked; suddenly only
  :3000 worked, no-port failed to load. nginx.conf itself was reviewed and
  is correctly configured (proxy_pass http://127.0.0.1:3000, proper
  X-Forwarded-* headers, 130s timeouts covering the Ollama 120s call
  timeout). `Get-Process nginx` showed nginx WAS running (2 worker
  processes). Root cause not fully isolated in this session, but the
  standard restart sequence (`nginx.exe -s stop` then `nginx.exe` again from
  C:\nginx) was given as the fix — nginx has no persistent state to lose so
  this is low-risk. After the restart, no-port access returned a real page
  (not a connection failure) — landed back on Issue #2's CORS/CLIENT_URL
  mismatch as the remaining problem, not nginx itself.
